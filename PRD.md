# KcalShot — 产品需求文档（PRD）

> 仓库名：`kcalshot` · 版本 v0.5 · 2026-06-24 · 统一健康评分口径、验收指标更可验收、明确 HealthKit 最低交付范围、needsReview 项数口径

---

## 1. 产品概述

一款 iOS 健康饮食管理 App。用户绑定**自己的 LLM API**，每天把一日三餐拍照导入 App，App 调用视觉大模型自动识别食物的**热量、营养成分（蛋白质/脂肪/碳水）和健康评分（1–10 分）**，从而帮助用户监控每日热量与营养摄入。

**一句话价值主张：** 拍张照，就知道这顿吃了多少卡、够不够健康，用你自己的 AI，数据流向你自己掌控。

### 1.1 核心差异点
- **自带 API（BYO-API）**：不依赖我们的后端，用户用自己的 key，成本与额度自控。
- **多模型可切换**：对接中转站时可预选多个模型，识别不准就一键换模型重识别。
- **无中间方**：App 自身无后端、不收集数据、除用户配置的 endpoint 外不发送任何图片。图片去向由用户自己选择的 API 决定（见 §4.1 数据与隐私说明），我们不在中间截留或转发。

## 2. 目标用户与场景

**主用户（产品围绕其设计）：健身/减脂人群。** 高频记录、诉求（每日热量+宏量营养素追踪）与产品闭环最契合。一切优先级以他们的体验为准。

| 角色 | 定位 | 场景 |
|------|------|------|
| 健身/减脂人群 | **主用户** | 每餐快速估算热量与宏量营养素，控制每日摄入 |
| 注重隐私 / 技术型用户 | **使能人群**（不是独立主用户，是"为何自带 API"的前提） | 已有 LLM API（OpenAI / 中转站 / 本地 Ollama），希望数据流向自控 |

**非目标用户**：慢病 / 临床营养管理人群。本产品输出为 **AI 估算**，不具备医疗级准确性，不用于诊断、治疗或临床营养决策；不针对该人群做专门设计，也不承担相应责任。

**典型流程**：早餐 → 拍照 → AI 识别 → （必要时修正/换模型）→ 保存到"早餐" → 一天结束查看总热量与营养达标情况。

## 3. 功能需求（MVP）

### F1. API 设置与多模型管理
- 配置**全局默认** OpenAI 兼容接口：`base_url`、`api_key`。
- 维护一个**模型清单**（可增删），每个模型含：显示名、model id、是否支持视觉、**可选的覆盖 base_url 与覆盖 key**。
  - 留空则继承全局默认；填了则该模型用自己的 endpoint/key。
  - 兼顾两种场景：多数模型共用同一中转站（用全局），个别模型接本地 Ollama / 专用代理（用覆盖）。
- 设置默认识别模型。
- "测试连接"按钮：对全局或单个模型发轻量请求验证可用。
- 所有 key（全局 + 模型覆盖）存入 **Keychain**，不明文落盘。

### F2. 拍照 / 相册识别
- 从相机拍摄或相册选择食物照片。
- 调用默认视觉模型，结构化返回：
  - 食物名称（可多项）
  - 总热量（kcal）
  - 宏量营养素：蛋白质 / 脂肪 / 碳水（g）
  - **健康评分：1–10 分**（10 = 最健康），并附等级标签（如 优/良/中/差）与简短理由
  - 置信度/说明
- 识别中显示进度；失败可重试。
- **一键换模型重识别**：对同一张照片切换模型重新分析。

### F3. AI 结果可手动修正
- 识别结果以可编辑表单呈现：用户可改名称、热量、各营养素数值、健康评分（1–10）。
- **允许直接保存，不强制修改**（降低每餐记录摩擦）；但当结果被标记为"建议核对"时（见 F3.1），默认展开编辑并高亮提示。

### F3.1 识别结果规则（置信度与不确定性处理）

LLM 自报的置信度**不是统计意义上的概率**，只是模型主观判断且通常偏乐观，因此只作"软信号"驱动提醒，不作真值、不向用户展示伪精确百分比。

**两维不确定性**（误差大头是"份量"而非"认错食物"，故拆开）：
- `recognitionConfidence` 0–1：模型自报的"认得准不准"。
- `portionAssumed` bool + `assumptions` 文本：份量未知时模型须按标准份量估算并**显式说明假设**（如"按一份约 250g 估算"）。

**客户端启发式合成 `needsReview`**（不依赖单一自报值）：
```
needsReview = recognitionConfidence < 0.7
           || portionAssumed == true
           || items.count > 2        // 混合餐逐项更易错
```
> `items.count` **以模型识别出的食物项数为准**（即返回 JSON `items[]` 的长度），不是配料数；一道混合菜若被模型作为单项返回则计 1，被拆成多项则按项计。避免单道菜/多配料带来的歧义。

**处理规则：**
- `needsReview == true`：UI 高亮"⚠︎ 建议核对份量/识别"，默认展开编辑，**但不强制**——用户仍可直接保存。
- `needsReview == false`：正常显示，可一键保存。
- **份量未知**：模型按标准份量估，写入 `assumptions`，字段可改。
- **混合菜**：支持逐项拆分（每项含名称/热量/营养）或单条聚合，用户可删改项。
- 全文将识别数值定性为**"估算"**，不承诺精确。
- **高精度手段后置**：多次采样自洽、多模型交叉作为 v1.1+ 的可选"高精度模式"，MVP 不做（避免 N 倍成本/延迟）。

### F4. 三餐记录
- 每条记录归类到 早餐 / 午餐 / 晚餐 / 加餐。
- 字段：日期、餐次、食物名、热量、蛋白/脂肪/碳水、健康评分（1–10）、备注、缩略图（本地）。
- 历史按日期浏览，可编辑/删除。

### F5. 每日营养统计
- 汇总当天：总热量、蛋白/脂肪/碳水合计。
- 与每日目标对比（进度条/环形图）。
- 当天平均健康评分概览（如各餐 1–10 分的均值与趋势）。

### F6. 目标设定（TDEE）
- 输入性别、年龄、身高、体重、活动水平 → 计算 BMR（Mifflin-St Jeor）与 TDEE。
- 给出推荐每日热量与三大营养素目标，**允许手动修改**。

### F7. HealthKit 同步
- 将每日摄入热量（Dietary Energy）、可选体重写入 Apple"健康"。
- 申请最小必要权限；用户可在设置中开关。
- **优先级**：MVP 范围内，但属里程碑末端（M5）。
- **首版最低交付范围（必做下限）**：经用户授权后，把**当日摄入总热量**写入 HealthKit 的 Dietary Energy（单向写、每日一次/记录变更时更新）；不写体重、不读取任何 HealthKit 数据。授权被拒时优雅降级、不影响其他功能。
- **可选增量（时间允许才做）**：写入体重、按餐次写入。
- **降级/后置条款**：若工期紧张，可仅交付上述"最低范围"，或整体后置到 v1.1——不阻塞核心闭环上线。

### F8. MVP 非目标（明确不做）

为控制首版范围，以下**明确不做**，避免范围膨胀：
- 条码扫描录入、食材/菜品本地数据库。
- 拍照以外的录入方式（手动搜索食物、语音录入）。
- 自动分餐（自动判断照片属于哪一餐，由用户选）。
- 多次采样 / 多模型交叉的"高精度模式"（见 F3.1，后置 v1.1+）。
- 社交分享、社区、教练/营养师对接。
- 任何形式的医疗/临床建议（见 §2 非目标用户）。
- 流式输出、自有云同步/账号体系。

## 4. 非功能需求

- **隐私**：见 §4.1 数据与隐私说明。App 无自有后端、无第三方分析 SDK。
- **离线**：除识别需联网外，记录与统计完全本地可用。
- **性能**：识别请求异步、不阻塞 UI；图片上传前压缩到合理分辨率（降低 token 成本与流量）。
- **健壮性**：API 报错、超时、JSON 解析失败均有清晰提示与重试（详见 §5.3）。
- **国际化**：界面**中英双语**，跟随系统语言切换（`Localizable.strings` / String Catalog）。识别 prompt 中要求模型用界面当前语言输出食物名称与理由。

### 4.1 数据与隐私说明

**诚实边界：** App 本身不设服务器、不收集、不分析、不转发用户数据。但**识别必然要把图片发往用户自己配置的 API endpoint**——若该 endpoint 是第三方服务（OpenAI / 中转站 / 代理），图片就会离开本机并按**该服务的条款**被处理。这部分的信任与责任边界在用户对 endpoint 的选择，App 无法也不代为担保。设置页与首次识别需向用户**明确告知此点**。

| 数据 | 存哪里 | 是否上传 | 说明 |
|------|--------|----------|------|
| 原始食物照片 | 不持久化 | **仅上传到用户配置的 endpoint** | 识别后不在本机长期保存原图 |
| 缩略图 | App 沙盒（SwiftData） | 否 | 仅供记录展示 |
| 识别结果 / 三餐记录 / 目标 | App 沙盒（SwiftData） | 否 | 本地 |
| API key（全局+覆盖） | **Keychain** | 否（仅作为请求头发往对应 endpoint） | 不明文落盘 |
| 每日热量 / 体重 | 本地 + 可选写入 HealthKit | 否 | HealthKit 由用户授权 |

**用户责任边界（需在 App 内呈现）：** 选择第三方 endpoint 即表示接受其数据处理条款；如需"绝不外流"，应使用本地模型（如 Ollama）作为 endpoint。

## 5. 技术架构

- **UI**：SwiftUI
- **数据**：SwiftData（最低部署目标 **iOS 17**；Xcode 16.2 支持）
- **架构**：MVVM；网络层独立封装
- **密钥**：Keychain（key），UserDefaults（非敏感设置）
- **健康**：HealthKit

### 5.1 目录结构（计划）
```
HealthFoodManager/
├── App/                    # 入口、ModelContainer
├── Models/                 # SwiftData @Model: MealEntry, DailyGoal, APIModelConfig
├── Services/
│   ├── LLMClient.swift     # OpenAI 兼容请求 + 视觉
│   ├── KeychainStore.swift
│   └── HealthKitManager.swift
├── Features/
│   ├── Capture/            # 拍照/识别/修正
│   ├── Diary/              # 三餐记录与历史
│   ├── Stats/              # 每日统计
│   └── Settings/           # API & 模型 & 目标
└── Shared/                 # 复用组件、扩展
```

### 5.2 数据模型（草案）
- `MealEntry`：id, date, mealType(早/午/晚/加餐), name, calories, protein, fat, carbs, healthScore(1–10), note, thumbnailData, modelUsed
- `DailyGoal`：targetCalories, protein, fat, carbs, gender, age, height, weight, activityLevel
- `APIModelConfig`：displayName, modelId, supportsVision, isDefault, **overrideBaseURL?（可空，空则用全局）**（key 与覆盖 key 均存 Keychain，按 id 索引）
- 全局设置（非 SwiftData，存 UserDefaults/Keychain）：globalBaseURL, globalKey, uiLanguage

### 5.3 LLM 接口设计
- `POST {base_url}/chat/completions`，base_url 取模型覆盖值，否则取全局默认；多模态 message：text 指令 + image_url（base64 data URI）。
- 用 **system prompt 约束输出为严格 JSON**，字段含：`items[]`（每项 名称/热量/蛋白/脂肪/碳水）、总计、`healthScore` 1–10、`reason`、`recognitionConfidence` 0–1、`portionAssumed` bool、`assumptions` 文本。要求按界面当前语言输出文本字段；客户端解析后按 F3.1 规则计算 `needsReview`。解析失败则重试一次，再失败提示并保留原始文本供手动录入。
- 图片压缩为 JPEG（约 ≤1024px 长边）后 base64。

**兼容端差异与健壮性处理（不同 OpenAI 兼容端行为差异大，需显式处理）：**
- **能力探测 / 视觉支持**：模型是否支持视觉以配置项 `supportsVision` 为准（用户标注）；可选用 `GET /models` 辅助填充模型列表，但不强依赖（部分端不实现）。**不支持视觉的模型禁止设为识别模型**，UI 置灰并提示。
- **流式 vs 非流式**：MVP **仅用非流式**（`stream:false`），一次性拿完整 JSON，解析最简单；流式留待后续。
- **超时与重试**：请求设超时（如 60s）；网络/5xx 错误**指数退避重试**（如最多 2 次）；4xx（鉴权/参数）不重试，直接提示。
- **JSON 失败回退**：优先要求 JSON-only 输出（可附 `response_format: json_object`，但不强依赖，因部分端不支持）；解析失败→重试 1 次；仍失败→提示并保留模型原始文本，供用户手动录入，不丢弃。
- **错误分类提示**：区分"key/鉴权错"、"endpoint 不可达"、"模型不存在/不支持视觉"、"返回无法解析"，给出对应可操作提示。

### 7.1 验收标准（每功能 ≥1 条可测指标）

> 注：上线前无法测日活/留存，故 MVP 验收用**任务级可测指标**。数值为草案目标，真机调优后修订。

| 功能 | 验收指标（草案） |
|------|------------------|
| F1 API/模型管理 | "测试连接"能在 ≤10s 内对有效/无效配置给出正确成败结论；key 仅存 Keychain（不出现在 UserDefaults/明文） |
| F2 拍照识别 | 对一批常见餐食样本，**结构化 JSON 解析成功率 ≥95%**；单次识别端到端 **P50 ≤8s、P90 ≤15s**（不含用户网络极端情况） |
| F3 结果修正 | 识别结果所有字段可编辑并正确保存；编辑后数值即时反映到当日统计 |
| F3.1 不确定性 | `needsReview` 触发条件命中时 100% 显示提示并默认展开编辑；低置信不阻断保存 |
| F4 三餐记录 | 保存/编辑/删除成功率 100%；记录正确归入对应日期与餐次 |
| F5 每日统计 | 当日合计 = 各条记录之和（核对一致）；与目标对比显示正确 |
| F6 TDEE | Mifflin-St Jeor 计算结果与公式核对一致；手动改目标后持久生效 |
| F7 HealthKit | 授权后每日热量正确写入"健康"；拒权时优雅降级不崩溃 |
| 全局健壮性 | 鉴权错/不可达/解析失败均给出可区分提示；**关键流程（识别→修正→保存、设置、统计）无已知崩溃，严重崩溃数为 0** |

## 8. 风险与对策

| 风险 | 对策 |
|------|------|
| LLM 热量估计不准 | 多模型切换 + 手动修正；明确告知为"估算" |
| 不同中转站返回格式不一致 | 严格 JSON prompt + 容错解析 + 测试连接 |
| 视觉模型差异（有的模型不支持图片） | 模型配置标注"是否支持视觉"，识别只用支持的 |
| 图片体积大、token 成本高 | 上传前压缩 |
| 隐私顾虑 / 责任边界 | §4.1 诚实说明：App 无中间方，但第三方 endpoint 的数据处理责任在用户；提供本地模型（Ollama）作为"绝不外流"路径 |
| 慢病用户误用为医疗工具 | 明确为 AI 估算、非医疗用途；§2 列为非目标用户 |
| HealthKit 拖累工期 | 置于 M5 末端，可降级最小版或后置 v1.1，不阻塞核心闭环 |
